Fractional Factorial Design

Incident Response Automation

Fractional factorial of 5 incident response parameters for MTTR and false escalation rate

Summary

This experiment investigates incident response automation. Fractional factorial of 5 incident response parameters for MTTR and false escalation rate.

The design varies 5 factors: alert threshold severity (level), ranging from 1 to 5, escalation delay min (min), ranging from 1 to 15, auto remediation enabled, ranging from off to on, runbook timeout sec (sec), ranging from 30 to 300, and notification channels (channels), ranging from 1 to 5. The goal is to optimize 2 responses: mttr min (min) (minimize) and false escalation pct (%) (minimize). Fixed conditions held constant across all runs include platform = pagerduty, integration = slack.

A fractional factorial design reduces the number of runs from 32 to 8 by deliberately confounding higher-order interactions. This is ideal for screening — identifying which of the 5 factors matter most before investing in a full study.

Key Findings

For mttr min, the most influential factors were notification channels (36.2%), alert threshold severity (26.5%), runbook timeout sec (20.5%). The best observed value was 5.2 (at alert threshold severity = 1, escalation delay min = 15, auto remediation enabled = on).

For false escalation pct, the most influential factors were escalation delay min (42.4%), runbook timeout sec (38.5%), alert threshold severity (17.2%). The best observed value was -0.6 (at alert threshold severity = 5, escalation delay min = 15, auto remediation enabled = off).

Recommended Next Steps

  • Follow up with a response surface design (CCD or Box-Behnken) on the top 3–4 factors to model curvature and find the true optimum.
  • Consider whether any fixed factors should be varied in a future study.
  • The screening results can guide factor reduction — drop factors contributing less than 5% and re-run with a smaller, more focused design.

Step-by-Step Workflow

1

Preview the design

Terminal
$ python doe.py info --config use_cases/85_incident_response_automation/config.json
2

Generate the runner script

Terminal
$ python doe.py generate --config use_cases/85_incident_response_automation/config.json \ --output use_cases/85_incident_response_automation/results/run.sh --seed 42
3

Execute the experiments

Terminal
$ bash use_cases/85_incident_response_automation/results/run.sh
4

Analyze results

Terminal
$ python doe.py analyze --config use_cases/85_incident_response_automation/config.json
5

Get optimization recommendations

Terminal
$ python doe.py optimize --config use_cases/85_incident_response_automation/config.json
6

Multi-objective optimization

With 2 competing responses, use --multi to find the best compromise via Derringer–Suich desirability.

Terminal
$ python doe.py optimize --config use_cases/85_incident_response_automation/config.json --multi
7

Generate the HTML report

Terminal
$ python doe.py report --config use_cases/85_incident_response_automation/config.json \ --output use_cases/85_incident_response_automation/results/report.html
