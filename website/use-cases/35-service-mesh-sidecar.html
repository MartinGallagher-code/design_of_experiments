Plackett-Burman Design

Service Mesh Sidecar

Plackett-Burman screening of 6 Envoy sidecar parameters for latency overhead and resource usage

Summary

This experiment investigates service mesh sidecar. Plackett-Burman screening of 6 Envoy sidecar parameters for latency overhead and resource usage.

The design varies 6 factors: concurrency (threads), ranging from 1 to 8, circuit breaker max (conns), ranging from 100 to 10000, access log, ranging from off to on, tracing sample pct (%), ranging from 1 to 100, compression, ranging from off to on, and connection pool (conns), ranging from 100 to 1000. The goal is to optimize 2 responses: latency overhead ms (ms) (minimize) and sidecar cpu pct (%) (minimize). Fixed conditions held constant across all runs include mesh = istio, protocol = http2.

A Plackett-Burman screening design was used to efficiently test 6 factors in only 8 runs. This design assumes interactions are negligible and focuses on identifying the most influential main effects.

Key Findings

For latency overhead ms, the most influential factors were concurrency (35.7%), tracing sample pct (16.7%), connection pool (16.0%). The best observed value was 0.5 (at concurrency = 1, circuit breaker max = 100, access log = on).

For sidecar cpu pct, the most influential factors were circuit breaker max (39.6%), compression (24.2%), tracing sample pct (17.6%). The best observed value was 0.5 (at concurrency = 8, circuit breaker max = 100, access log = off).

Recommended Next Steps

  • Follow up with a response surface design (CCD or Box-Behnken) on the top 3–4 factors to model curvature and find the true optimum.
  • Consider whether any fixed factors should be varied in a future study.
  • The screening results can guide factor reduction — drop factors contributing less than 5% and re-run with a smaller, more focused design.

Step-by-Step Workflow

1

Preview the design

Terminal
$ python doe.py info --config use_cases/35_service_mesh_sidecar/config.json
2

Generate the runner script

Terminal
$ python doe.py generate --config use_cases/35_service_mesh_sidecar/config.json \ --output use_cases/35_service_mesh_sidecar/results/run.sh --seed 42
3

Execute the experiments

Terminal
$ bash use_cases/35_service_mesh_sidecar/results/run.sh
4

Analyze results

Terminal
$ python doe.py analyze --config use_cases/35_service_mesh_sidecar/config.json
5

Get optimization recommendations

Terminal
$ python doe.py optimize --config use_cases/35_service_mesh_sidecar/config.json
6

Multi-objective optimization

With 2 competing responses, use --multi to find the best compromise via Derringer–Suich desirability.

Terminal
$ python doe.py optimize --config use_cases/35_service_mesh_sidecar/config.json --multi
7

Generate the HTML report

Terminal
$ python doe.py report --config use_cases/35_service_mesh_sidecar/config.json \ --output use_cases/35_service_mesh_sidecar/results/report.html
