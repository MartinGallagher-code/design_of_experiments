Box-Behnken Design

Feature Flag Evaluation

Box-Behnken design to tune cache TTL, rule complexity, and SDK polling interval for evaluation latency and cache hits

Summary

This experiment investigates feature flag evaluation. Box-Behnken design to tune cache TTL, rule complexity, and SDK polling interval for evaluation latency and cache hits.

The design varies 3 factors: cache ttl sec (sec), ranging from 5 to 120, rule complexity score (score), ranging from 1 to 10, and sdk polling interval sec (sec), ranging from 10 to 300. The goal is to optimize 2 responses: evaluation latency us (us) (minimize) and cache hit rate pct (%) (maximize). Fixed conditions held constant across all runs include platform = launchdarkly, sdk = server_side.

A Box-Behnken design was chosen because it efficiently fits quadratic models with 3 continuous factors while avoiding extreme corner combinations — requiring only 15 runs instead of the 8 needed for a full factorial at two levels.

Quadratic response surface models were fitted to capture potential curvature and factor interactions. The RSM contour plots below visualize how pairs of factors jointly affect each response.

Key Findings

For evaluation latency us, the most influential factors were cache ttl sec (48.7%), sdk polling interval sec (28.4%), rule complexity score (22.8%). The best observed value was 115.0 (at cache ttl sec = 120, rule complexity score = 5.5, sdk polling interval sec = 10).

For cache hit rate pct, the most influential factors were cache ttl sec (56.0%), rule complexity score (24.5%), sdk polling interval sec (19.6%). The best observed value was 90.6 (at cache ttl sec = 120, rule complexity score = 5.5, sdk polling interval sec = 10).

Recommended Next Steps

  • Run confirmation experiments at the predicted optimal settings to validate the model.
  • Consider whether any fixed factors should be varied in a future study.

Step-by-Step Workflow

1

Preview the design

Terminal
$ python doe.py info --config use_cases/84_feature_flag_evaluation/config.json
2

Generate the runner script

Terminal
$ python doe.py generate --config use_cases/84_feature_flag_evaluation/config.json \ --output use_cases/84_feature_flag_evaluation/results/run.sh --seed 42
3

Execute the experiments

Terminal
$ bash use_cases/84_feature_flag_evaluation/results/run.sh
4

Analyze results

Terminal
$ python doe.py analyze --config use_cases/84_feature_flag_evaluation/config.json
5

Get optimization recommendations

Terminal
$ python doe.py optimize --config use_cases/84_feature_flag_evaluation/config.json
6

Multi-objective optimization

With 2 competing responses, use --multi to find the best compromise via Derringer–Suich desirability.

Terminal
$ python doe.py optimize --config use_cases/84_feature_flag_evaluation/config.json --multi
7

Generate the HTML report

Terminal
$ python doe.py report --config use_cases/84_feature_flag_evaluation/config.json \ --output use_cases/84_feature_flag_evaluation/results/report.html
